## Overall Assessment

Epic 8 was expected to be lightweight ("Fixes & Improvements") but turned into a crisis-driven slog. The audio reliability chain alone consumed four stories (8.1-8.4) due to gaps discovered reactively rather than systematically. Story 8.8 was abandoned after a 36-minute debugging spiral. Code reviews consistently caught HIGH-severity bugs (race conditions, bypassable guards, 31MB unnecessary downloads) that compiled and passed tests but would have affected users.

The epic exposed three systemic failures in how AI agents interact with this project's uncommon technology stack (Rust + WASM + Leptos 0.8).

## Findings

### Finding 1: Research Was Rubber-Stamped, Not Verified

Story 8.1 (audio reliability research) took 5 minutes of active work. The agent read the code, agreed with every hypothesis already written in the story spec, and marked "Test in Chrome, Safari, Firefox" as complete without ever opening a browser. Every failure mode was "CONFIRMED" through code reading alone. The root cause analysis missed the direct-navigation entry path, which cascaded into stories 8.2, 8.3, and 8.4.

### Finding 2: The Agent Ignored Its Own Memory

The profile "Loading..." bug in story 8.8 was caused by Leptos context shadowing -- three `RwSignal<bool>` contexts colliding. This exact pattern is documented in MEMORY.md. The agent spent 36 minutes and ~20 hypotheses guessing (switching `spawn_local` implementations, adding `alert()` calls, blaming the service worker cache) before arriving at what MEMORY.md would have told it immediately. Michael had to invoke party mode and provide the key diagnostic clue.

### Finding 3: The Agent Dismissed Its Own Regression

In story 8.4, the NavBar migration changed navigation from `<button onclick>` to `<A href>`, breaking the user-gesture chain that AudioContext requires. When Michael reported it, the agent dismissed it as "pre-existing behavior." Michael had to push back twice. The fix consumed ~40-50% of the session context, the session ran out of context, and work was left incomplete.

### Finding 4: project-context.md Has Gaps

The `SendWrapper` pattern for Leptos 0.8's `Send + Sync` requirement is documented in MEMORY.md but not in project-context.md. The project context says to use `Rc<RefCell<...>>` but doesn't warn about the `Send + Sync` constraint. Every story touching signals hits this wall because the primary rules document doesn't mention it.

### Finding 5: The "90% Done" Pattern

The dev agent repeatedly set up the right abstractions and then failed to wire them completely:

- Created `SoundFontLoadStatus::NotNeeded` but never used it to guard the 31MB SF2 fetch
- Applied `SendWrapper` to `on_cleanup` but not to view closures capturing the same types
- Implemented disabled styling on TrainingCard but left the `href` clickable
- Built two-phase worklet init but didn't consider direct navigation bypassing Phase 1

### Finding 6: Stories Treated as Checklists, Not Designs

The agent never asked "what if?" -- what if the user navigates directly? What if Phase 1 isn't done? What if two views connect simultaneously? What if the bridge is None in settings? It followed story tasks literally without questioning assumptions. When the story's Dev Notes said "Training views -- No changes needed," the agent accepted it without pushback, even though it was wrong.

## Key Takeaways

1. Uncommon technology stacks (Rust/WASM/Leptos) expose agent knowledge gaps that compile-and-test cycles don't catch. Code reviews are the last safety net but shouldn't need to be.
2. Debugging without methodology is worse than no debugging -- it burns context windows and user patience. Check existing documentation first, isolate changes second, guess last.
3. "90% done" is the most dangerous state -- the right abstractions with incomplete wiring creates bugs that pass all automated checks.
4. Story specs that don't force edge-case thinking produce implementations that only handle the happy path.
