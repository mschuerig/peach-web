**Research Scope 2: State Change Handler**
5. Both views register `onstatechange` handlers (pitch_comparison_view.rs:379-398, pitch_matching_view.rs:466-484) that call `interrupt_and...`

The state change handler fires immediately if the AudioContext starts suspended, interrupting training before any audio plays, and there's no attempt to resume the context first. The worklet initialization in app.rs:162 calls `get_or_create()` inside `spawn_local()` — outside any user gesture context — so if this runs before the user navigates to training, the AudioContext gets created suspended on Chrome. When the training view later calls `get_or_create()`, it retrieves that already-suspended context. The SoundFont fallback in note_player.rs falls back to oscillators if the worklet bridge isn't available.

Both training views log errors but continue silently without any user-visible indication that playback failed. Now I'm organizing the remaining work: I need to add diagnostic logging first, then write the root cause analysis and mitigation strategy, and update the story file to mark the completed research tasks.

**Story 8.1: Audio Playback Reliability Research** is complete and ready for review.

**Key findings:**

1. **Primary root cause:** `init_worklet_bridge()` in `app.rs` creates the AudioContext via `spawn_local()` outside any user gesture. On Chrome, this means the context starts `suspended`. When the training view later calls `get_or_create()`, it reuses the suspended context. Since `resume()` is never called, all audio output is silently dropped.

2. **Why it's intermittent:** It's a race condition — if the SoundFont fetch is slow and hasn't created the AudioContext yet when the user starts training, the training view creates it within the gesture context and audio works. Fast network/cached assets = silent failure.

3. **Secondary issues:** The `onstatechange` handler aggressively interrupts on any suspension (no recovery attempt), and playback errors are logged but invisible to the user.

**Top recommended fixes:**
- Add `AudioContext.resume()` call at training start (quick win, fixes most cases)
- Soften the `onstatechange` handler to attempt recovery before interrupting
- Restructure worklet init to not create AudioContext eagerly (proper architectural fix)
