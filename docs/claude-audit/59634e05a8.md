## Story 5.2 Validation — Architectural Focus

### Overall Assessment: **Good with notable risks**

The story is well-structured, correctly identifies the associated-type constraint, and proposes a sound enum-wrapper pattern. The task decomposition is logical. However, there are several architectural concerns that should be addressed before implementation.

---

### Finding 1: Synchronous Audio Rendering Blocks the Main Thread (HIGH)

The `play()` and `play_for_duration()` methods render audio by calling `synth.read_next()` in a loop — 44,100 iterations per second of audio. For a 1-second note, that's ~44K iterations on the main (and only) thread. In WASM this could take 10-50ms, causing a visible UI micro-stutter.

The story acknowledges this in "Performance Considerations" but offers no concrete mitigation. Since comparison mode plays two notes back-to-back (reference + target), the worst case is two render stalls in quick succession.

**Recommendation:** Add a subtask to benchmark rendering time. If >16ms, consider pre-rendering common notes at startup or rendering in smaller chunks via `setTimeout`/`requestAnimationFrame` yielding.

---

### Finding 2: Indefinite Playback Duration for `play()` Is Unspecified (HIGH)

The `NotePlayer::play()` method returns a handle for **indefinite** playback (used in pitch matching — the user's tunable note plays until they commit). But `SoundFontNotePlayer` renders a fixed-length `AudioBuffer` upfront. The story never specifies how long this buffer should be.

Options:
- Render a "long enough" buffer (e.g., 30 seconds) — wasteful, delays onset
- Set `AudioBufferSourceNode.loop = true` — loops seamlessly but may have audible seam artifacts
- Render a moderate buffer (e.g., 5s) with looping

**Recommendation:** Task 2.3/2.4 should explicitly address the indefinite-play case. Looping an AudioBufferSourceNode with a sustain-phase buffer seems most practical.

---

### Finding 3: OxiSynth WASM Compatibility Is Unverified (MEDIUM)

oxisynth 0.1.0 has no explicit WASM support documentation. Its dependency chain includes:
- `soundfont` 0.1.0 — file parsing, likely fine
- `lewton` (optional) — Vorbis decoding, likely fine 
- `getrandom` (optional) — **known WASM footgun** without `wasm_js` feature

The story already has `getrandom` with `wasm_js` feature in `web/Cargo.toml`, but if oxisynth pulls in its own `getrandom` without that feature, compilation will fail. Also, oxisynth uses `rand` which depends on `getrandom`.

**Recommendation:** Add a spike subtask to Task 1: add the dependency and run `trunk build` before writing any code. If it fails, you'll need to patch `getrandom` features via `[patch]` or Cargo feature unification.

---

### Finding 4: Shared Synth + Program Switching Is Fragile (MEDIUM)

The story proposes a single `Rc<RefCell<Synth>>` shared via context, with `select_program()` called in the factory when creating a player. But `select_program` mutates the Synth's global channel state. If the user changes sound source mid-session, the shared Synth's program changes, affecting any subsequent rendering.

This works for the current single-user-single-view architecture, but it's implicitly coupled: the factory function has a side effect (changing the Synth's program) that isn't visible from its signature.

**Recommendation:** Document this coupling explicitly in the code. The factory should own the `select_program` call and the Synth should be treated as "last-writer-wins" for program selection.

---

### Finding 5: Architecture Doc Mentions `SoundFontStatus` Signal — Story Omits It (LOW)

The architecture doc (`architecture.md:383`) specifies:
```
soundfont_status: RwSignal<SoundFontStatus> (Loading, Ready, Failed)
```

The story simplifies to `Option<Rc<RefCell<Synth>>>` where `None` = loading or failed. Since the story explicitly says "no loading indicator" and "no error shown," this simplification is defensible, but it deviates from the architecture doc.

**Recommendation:** Acceptable simplification. Add a brief note in dev notes explaining why the tri-state signal was simplified to Option.

---

### Finding 6: 31 MB SoundFont Size vs Story's <5 MB Guidance (LOW)

Architecture doc line 201 mentions "31 MB file loaded async via `fetch()`." Task 1.3 correctly says "ideally < 5 MB." These are contradictory. The story's guidance is better — 31 MB is excessive for web delivery.

**Recommendation:** Already handled correctly in the story. No action needed.

---

### Finding 7: `play_for_duration` Return Type Mismatch (LOW)

The `NotePlayer::play_for_duration()` trait returns `Result<(), AudioError>` — no handle. But the story's Task 2.4 says "schedule stop via AudioBufferSourceNode (finite buffer = natural stop)." This is correct — the finite buffer handles its own stop. However, `stop_all()` needs to be able to cancel an in-progress `play_for_duration` note early (comparison view does this at `comparison_view.rs:425`). The story should ensure `SoundFontNotePlayer` tracks active source nodes for `stop_all()` like `OscillatorNotePlayer` tracks active handles.

**Recommendation:** Task 2.3 should mention tracking active `AudioBufferSourceNode`s internally for `stop_all()` cleanup, mirroring the `active_handles` pattern in `OscillatorNotePlayer`.

## Key Findings

**OxiSynth is designed for exactly what you want.** Its `read_next()` / `write_f32()` is a pull-based API: send NoteOn, keep pulling samples, send NoteOff, pull the release tail. The synth holds the note until told to stop. No pre-rendering needed.

**OxiSynth was built with WASM in mind** — the repo README says so explicitly, and there's a [live browser demo](https://oxisynth.netlify.app). It's pure Rust, no system deps, compiles cleanly to `wasm32-unknown-unknown`.

**The right architecture is AudioWorklet, not pre-rendered AudioBuffer.** The pattern:

```
Main Thread (Leptos)              Audio Thread (AudioWorklet)
┌──────────────────┐              ┌───────────────────────────┐
│ UI events         │  postMessage │ processor.js (~30 lines)  │
│ - NoteOn/Off     ─┼─────────────> - OxiSynth WASM instance  │
│ - SoundFont bytes ─┼─────────────> - Synth.write_f32() in   │
│ - adjust_frequency─┼─────────────>   process() callback     │
└──────────────────┘              └───────────────────────────┘
```

This solves everything:
- **No main thread blocking** — synth runs on the audio thread
- **Indefinite note duration** — synth sustains until NoteOff, naturally
- **Real-time pitch adjustment** — send pitch bend events via postMessage
- **Release tails** — handled by the synth automatically
- **<50ms onset** — AudioWorklet processes 128 samples at a time (~2.7ms per chunk)

**The catch:** `web-sys` can't subclass `AudioWorkletProcessor`. Every Rust+AudioWorklet project uses a small JS glue file (~30 lines) for the processor, with WASM compiled separately and loaded into the worklet via `postMessage`. This is the standard pattern.

## What This Means for Story 5.2

The story needs a significant rewrite. The "pre-render AudioBuffer" approach, the `UnifiedNotePlayer` enum wrapper, the `.detune` pitch adjustment — all of that is built on the wrong foundation. The architecture should be:

1. **Separate WASM build** for the AudioWorklet synth (or a `#[no_mangle]` export layer)
2. **Small JS AudioWorkletProcessor** that pulls samples from OxiSynth
3. **Message-passing interface** between main thread and worklet (NoteOn/Off, SoundFont loading, pitch control)
4. **`SoundFontNotePlayer`** on the main thread just sends messages to the worklet — it doesn't touch audio samples at all

Want me to rewrite the story with this corrected architecture, or would you prefer to do a deeper technical spike first (e.g., get a minimal OxiSynth AudioWorklet proof-of-concept running)?

Story 5.2 has been rewritten. Here's a summary of the key architectural changes:

**Core change:** Pre-rendered AudioBuffer → real-time AudioWorklet synthesis

| Aspect | Old approach | New approach |
|--------|-------------|--------------|
| Where synth runs | Main thread (blocking) | AudioWorklet audio thread |
| Note duration | Pre-rendered fixed buffer | Synth holds note until NoteOff |
| Pitch adjustment | `.detune` on AudioBufferSourceNode | MIDI pitch bend via postMessage |
| Communication | Direct Rust function calls | Message passing (main ↔ worklet) |
| WASM build | Single Trunk build | Trunk + separate `synth-worklet` crate |

**New structural elements:**
- `synth-worklet/` crate — `#[no_mangle]` C-style exports wrapping OxiSynth, compiled to raw WASM for the worklet
- `synth-processor.js` — ~50-line AudioWorkletProcessor that instantiates the WASM synth and pulls samples in `process()`
- `WorkletBridge` — main-thread adapter that wraps the MessagePort for sending NoteOn/Off/PitchBend to the worklet
- `SoundFontNotePlayer` now sends messages instead of touching audio samples

**What stayed the same:** UnifiedNotePlayer enum wrapper pattern, frequency-to-MIDI conversion, oscillator fallback, settings wiring, and the overall task flow (init → player → views → settings → test).
