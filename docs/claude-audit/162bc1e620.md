**As a** developer,
**I want** the perceptual profile (Welford's algorithm) and Kazez adaptive algorithm implemented and tested,
**So that** the adaptive training logic is verified before connecting it to UI and audio.

**Acceptance Criteria:**

**Given** an empty PerceptualProfile
**When** I call `update(C4, 50.0, true)`
**Then** the PerceptualNote for C4 has sampleCount=1 and mean=50.0

**Given** a profile with multiple samples for a note
**When** I check stdDev
**Then** it matches Welford's algorithm: `sqrt(m2 / (sampleCount - 1))`

**Given** a profile with trained and untrained notes
**When** I call `weakSpots(10)`
**Then** untrained notes rank highest (infinite score) and trained notes rank by descending mean threshold

**Given** a correct answer at p=50 cents
**When** `kazez_narrow(50.0)` is called
**Then** it returns approximately 32.3 cents (`50 * (1.0 - 0.05 * sqrt(50))`)

**Given** an incorrect answer at p=50 cents
**When** `kazez_widen(50.0)` is called
**Then** it returns approximately 81.8 cents (`50 * (1.0 + 0.09 * sqrt(50))`)

**Given** no lastComparison and no overallMean (cold start)
**When** `next_comparison()` is called
**Then** magnitude starts at maxCentDifference (100 cents)

**Given** a profile with overallMean=25.0 and no lastComparison
**When** `next_comparison()` is called
**Then** magnitude starts at 25.0 cents (profile-informed start)

### Story 1.5: ComparisonSession State Machine

**As a** developer,
**I want** the ComparisonSession state machine working with the observer pattern,
**So that** the complete training loop runs correctly: generate → play → answer → broadcast → feedback → next.

**Acceptance Criteria:**

**Given** an idle ComparisonSession
**When** `start()` is called with intervals containing prime (unison)
**Then** the state transitions to `playingNote1` and the first reference note plays

**Given** state is `playingNote2` (early answer allowed)
**When** `handle_answer(true)` is called
**Then** the target note stops immediately, observers receive `comparison_completed` with a correct CompletedComparison, and state transitions to `showingFeedback` (FR4)

**Given** state is `awaitingAnswer`
**When** `handle_answer(false)` is called
**Then** observers receive `comparison_completed` with an incorrect CompletedComparison

**Given** state is `showingFeedback`
**When** 400ms elapses
**Then** the next comparison begins automatically (loop continues)

**Given** a running session
**When** `stop()` is called
**Then** all audio stops, async tasks are cancelled, state returns to `idle`, and the incomplete comparison is discarded (FR6, FR7)

**Given** the session is running
**When** multiple comparisons complete in sequence
**Then** the adaptive algorithm (KazezNoteStrategy) adjusts difficulty based on correctness (FR8)

### Story 1.6: Comparison Record Persistence & Profile Hydration

**As a** user,
**I want** my training records to survive page refreshes and browser restarts,
**So that** my training progress is never lost and the algorithm always knows my ability level.

**Acceptance Criteria:**

**Given** a completed comparison
**When** the ComparisonObserver fires
**Then** a ComparisonRecord is saved to IndexedDB `comparison_records` object store (FR36)

**Given** stored comparison records exist
**When** the app launches
**Then** all records are fetched (sorted by timestamp) and the PerceptualProfile is hydrated by replaying each record (FR21)

**Given** profile hydration with 100 records
**When** hydration completes
**Then** the profile state is identical to what it would be if 100 comparisons were completed in sequence (NFR9)

**Given** a storage write failure
**When** saving a comparison record fails
**Then** the user is informed that data may not have been saved (NFR8)

**Given** settings values (noteRangeMin, noteRangeMax, noteDuration, referencePitch, tuningSystem, varyLoudness)
**When** set or changed
**Then** they persist to localStorage with `peach.*` prefix keys and survive page refresh (FR30, FR38)

**Given** the app launches with existing localStorage settings
**When** the UserSettings adapter reads them
**Then** previously saved values are returned (not defaults)

**Given** the user answers incorrectly
**When** feedback appears

**Given** training is active

**Given** the browser tab is hidden or AudioContext is suspended
**When** the user returns
**Then** they are on the start page — training was stopped silently (FR48)

**Given** the UIObserver bridge
**When** domain observers fire (comparison completed, state change)
**Then** Leptos RwSignals update and the UI re-renders affected components
