## 🔴 HIGH ISSUES

### H1: Interval hrefs are computed once at render, not reactively (start_page.rs:42-43)

`interval_href()` calls `LocalStorageSettings::get_selected_intervals()` once when `StartPage` renders. If the user navigates to Settings, changes their selected intervals, and returns to the start page **without a full page reload**, the interval links will still point to the old interval selection.

In the old code, `navigate_with_intervals()` was called inside an `on:click` closure, so it always read the latest settings at click time. The new `<a href>` approach bakes the URL in at render time.

**Fix:** Either make the href reactive (compute inside a `move ||` closure or use a derived signal), or re-read settings in an `on:click` handler that calls `window().location().set_href()`.

---

## 🟡 MEDIUM ISSUES

### M1: `<nav>` landmark removed — training cards lost navigation semantics (start_page.rs:50)

The old code wrapped training buttons in `<nav aria-label="Training modes">`. The new code uses a plain `<div>` wrapper. Screen readers will no longer announce the training cards as a navigation region. The `<section>` elements help, but the overall navigation landmark is missing.

**Fix:** Wrap the outer `<div class="flex w-full flex-col gap-7 ...">` in a `<nav aria-label="Training modes">` or make the `<div>` itself a `<nav>`.

### M2: Dead module declaration — `mod profile_preview` with no public exports (mod.rs:6)

`profile_preview` is declared as a module but has no `pub use` and no remaining consumers. The story says "keep the file for story 7.4" but the module declaration without any public exports generates dead code. Since clippy doesn't flag it (the module has a public function internally), it's not technically broken, but it's unnecessary coupling.

**Severity reduced** because the story explicitly says to keep the file. Note for awareness.

### M3: Sprint status set to `review` but not `done` (sprint-status.yaml)

The commit sets `7-3-start-screen-redesign: review` in sprint-status.yaml, but per the workflow, if the review approves the story the status should be updated to `done`. This is expected to be updated at the end of this review — flagging for tracking.
