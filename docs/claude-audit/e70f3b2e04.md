Now I need to build three key pieces: the session module structure, the ComparisonSession state machine itself, and the observer traits that will handle state transitions.

The tricky part is reconciling the architecture with the domain crate's constraints — the training loop needs to be async with play, sleep, and wait_for_answer steps, but the domain crate can't depend on browser APIs. The solution is to express the session logic as async functions in the domain layer while keeping the actual timer and browser integration in the web crate, using a cancellation flag pattern to handle stopping between await points.
